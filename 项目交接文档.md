# MOOER 客服系统项目交接文档

交接日期：2026-06-29

## 1. 项目定位

本项目目前已经从“AI 自动回邮件脚本”演进为一个本地客服工作台原型，核心目标是：

- 同步客服邮箱邮件，保存到本地 SQLite 数据库。
- 使用 AI 做邮件分类、产品识别、Issue 归并和英文回复草稿生成。
- 在客服人工审核后，从邮箱草稿箱发送或删除草稿。
- 将同类售后反馈归并到 Issue Queue，支持研发报告导出和最终回复模板回填。
- 建立说明书、下载入口、保修政策、配件价格、回复模板、已解决 Issue 等知识库分层。

当前系统仍应视为“人工审核优先”的客服辅助系统，不建议无人值守自动发送。

## 2. 当前代码结构

### 2.1 核心入口

| 文件/目录 | 作用 |
| --- | --- |
| `email_automation.py` | 邮件自动处理主流程：读未读邮件、预检、AI 分类、生成草稿、写库、保存邮箱草稿。 |
| `dashboard.py` | Streamlit 客服控制台，目前是主要可用 UI。 |
| `api/` | FastAPI 后端，提供邮件、Issue、草稿、模板、知识库、日志、自动化控制等接口。 |
| `database.py` | SQLite 数据库封装，负责表初始化、邮件/Issue/模板/知识库读写。 |
| `ai_handler.py` | AI 分类、意图识别、产品识别和相关 prompt 逻辑。 |
| `response_generator.py` | 英文回复草稿生成，整合模板、说明书、知识库、最终回复模板。 |
| `imap_handler.py` | IMAP/SMTP 邮箱访问，读邮件、读草稿、保存/发送/删除草稿。 |
| `content_extractor.py` | 邮件正文清洗、当前内容提取、附件/引用处理。 |
| `email_analyzer.py` | Issue 查询、AI 分析和统计辅助。 |
| `issue_scanner.py` | Issue 扫描、日报同步、候选邮件归并相关脚本。 |
| `knowledge_base.py` | 知识库同步和分块索引逻辑。 |
| `part_price_db.py` | 配件价格初始化/维护工具。 |
| `pdf_reader.py` / `build_manual_cache.py` | 说明书 PDF 文本提取和缓存。 |

### 2.2 API 模块

FastAPI 入口：`api/main.py`

主要接口前缀：`/api/v1`

| Router | 功能 |
| --- | --- |
| `/emails` | 邮件列表、搜索、详情、线程上下文、状态/AI 分析更新。 |
| `/issues` | Issue Queue 列表、详情、候选复核、手动扫描、自动检测。 |
| `/analysis` | AI 问题查询、Issue 匹配、统计分析。 |
| `/automation` | 手动触发处理、查看调度状态、开关调度、修改间隔。 |
| `/drafts` | 邮箱草稿读取、发送、删除。 |
| `/logs` | 系统日志查看。 |
| `/prices` | 配件价格 CRUD。 |
| `/templates` | 回复模板 CRUD。 |
| `/knowledge` | 知识库统计、文档列表、同步。 |

API 文档地址：`http://localhost:8100/docs`

## 3. 运行方式

### 3.1 Python 环境

当前启动脚本使用的环境：

```text
C:\Users\USER\.workbuddy\binaries\python\envs\mooer-api
```

主要依赖见 `requirements.txt`，包括邮件处理、AI、PDF 读取、FastAPI 后端和 Streamlit 控制台所需依赖。

### 3.2 配置文件

| 文件 | 说明 |
| --- | --- |
| `.env` | 真实邮箱、OpenAI、第三方服务密钥等敏感配置。本文件不应提交。 |
| `config.yml` | 本地真实运行配置。本文件不应提交。 |
| `config.yml.sample` | 可提交的示例配置。 |

注意：仓库历史中的旧 `.env.sample` 曾包含真实邮箱密码，当前工作区已删除该文件。建议交接后立刻轮换该邮箱授权码/密码，并避免恢复旧 `.env.sample`。

### 3.3 启动命令

启动 API：

```bat
启动API.bat
```

或：

```bat
run_api.bat
```

启动 Streamlit 控制台：

```bat
启动控制台.bat
```

启动 API + Streamlit：

```bat
启动全部.bat
```

当前 `启动React控制台.bat` 和 `启动正式控制台.bat` 已写好 React 控制台启动逻辑，但工作区没有 `web/` 目录，所以 React 控制台还未真正落地。

## 5. 已落地能力

### 5.1 邮件处理闭环

- IMAP 读取未读邮件。
- 数据库去重和状态流转。
- 邮件预检，识别内部转发/疑似正在处理/高风险场景。
- AI 分类字段已落库，包括 `mail_category`、`issue_category`、`classification_confidence`、`classification_evidence`、`issue_fingerprint` 等。
- 生成英文回复草稿并保存到邮箱草稿箱。
- 失败邮件记录 `last_error` 和处理次数。

### 5.2 控制台

- 草稿按疑似 Issue 分组。
- Issue Queue 列表、筛选、详情。
- Issue 候选邮件人工复核。
- Issue 状态、研发状态、研发备注、解决方案、最终回复模板回填。
- 研发报告导出：HTML / Excel / 用户邮箱 TXT。
- 跳过邮件恢复。
- 知识库统计和同步。
- 配件价格管理。
- 回复模板管理。
- 系统日志查看。

### 5.3 Issue Queue

## 6. 当前主要风险

### 6.1 敏感数据风险

- `.env`、`config.yml`、`mooer_support.db`、数据库备份、日报、导出报告都可能包含真实邮箱、客户邮件正文、订单信息或内部处理记录。
- 旧 Git 历史中的 `.env.sample` 曾包含真实邮箱密码，建议立即轮换。
- 当前 `.gitignore` 已补充忽略 `.workbuddy/`、`manuals_cache/`、`Daily Report/`、`exports/`、数据库 WAL/SHM 和备份库。

### 6.2 自动处理风险

- 系统仍会默认在 API 启动后每 60 秒运行一次 `email_automation.py --once`。
- 目前设计上仍应只生成草稿，不建议直接自动发送。
- 高风险场景如退款、法律威胁、强烈投诉、固件失败导致设备不可用等必须人工复核。

### 6.3 数据状态风险

- 当前 `emails` 中仍有 376 封 `new`、120 封 `human_review`、16 封 `processing`。
- `processing` 如果长期不变，可能是旧运行残留状态，需要后续人工核查。
- 49 个 Issue 中 43 个仍是 `new_detected`，说明 Issue Queue 已能检测，但大量问题尚未被客服确认/合并/关闭。

### 6.4 前端迁移风险

- 启动脚本已经预留 React 控制台，但 `web/` 不存在。
- 正式控制台迁移还未开始，当前生产使用仍依赖 Streamlit。

### 6.5 依赖记录风险

- `requirements.txt` 已补入 Streamlit，但仍建议新同事用当前 `.workbuddy` 环境对照一次 `pip freeze`，确认是否还有间接依赖需要固定版本。
- 部分启动脚本写死了本机 Python 环境路径，迁移到新电脑时需要修改。

## 7. 建议继续开发事项

### P0：交接后立即处理

1. 轮换邮箱授权码/密码，避免历史泄露风险继续存在。
2. 清理长期停留在 `processing` 的邮件状态。
3. 对照当前环境 `pip freeze` 复核 `requirements.txt`，确认是否需要固定更多版本。
4. 对 `new`、`human_review`、`failed_retry` 邮件做一次人工/批量复核。
5. 逐个确认 `new_detected` Issue，合并重复项，排除误报，标记优先级。
6. 明确 GitHub 中哪些数据文件允许提交；默认不要提交数据库、日报、导出、缓存和日志。

### P1：稳定客服闭环

1. 完善最终回复模板优先机制：同类 Issue 已有最终模板时，后续邮件直接生成最终模板草稿。
2. 强化草稿与原始邮件匹配：优先 `Message-ID`、`In-Reply-To`、`References`，再回退到用户邮箱和规范化主题。
3. 完善历史邮件补分类脚本，不覆盖人工修正字段。
4. 给模板命中增加日志：记录模板 ID、命中原因、邮件 ID。
5. 将错误/停用模板彻底排除在 AI 使用范围外。

### P2：提高稳定性和可维护性

1. 引入 IMAP IDLE，保留 60 秒轮询兜底。
2. 优化 Streamlit 控制台性能，减少每次刷新读取邮箱/API 的阻塞。
3. 为数据库增加迁移脚本或版本记录，避免只靠 `CREATE TABLE IF NOT EXISTS` 和散落的 `ALTER TABLE`。
4. 增加最小自动化测试：数据库初始化、API schema、Issue 匹配、模板选择、知识库检索。
5. 将日志表定期归档，否则 `logs` 会持续膨胀。

### P3：正式控制台

1. 新建 React + Vite + TypeScript 前端。
2. 优先迁移 Issue Queue、Issue 详情、研发报告导出、邮件草稿审核。
3. 再迁移回复模板、配件价格、知识库、系统日志。
4. Streamlit 保留为备用后台，直到 React 控制台稳定。
