# 售后邮件处理与研发汇报开发计划

## 1. 当前目标

当前最重要的目标不是“AI 自动回复越多越好”，而是把售后邮件处理成一个稳定闭环：

1. 新邮件进来后能尽快进入系统。
2. AI 先分类、生成草稿，但不直接跳过高风险邮件。
3. 同类产品问题自动归并到 issue queue。
4. 客服可以一键查看相关用户邮箱、证据片段、邮件原文和影响范围。
5. 能导出结构化报告给研发，而不是一封封转发。
6. 研发给方案后，客服在 issue queue 填最终回复模板。
7. 后续同类邮件自动优先使用最终模板，并继续关联到同一个问题。

## 2. 当前主要问题

### 2.1 邮件处理链路仍不够稳

- 目前是 60 秒轮询 IMAP，不是真正实时推送。
- 邮箱草稿与数据库原始邮件匹配曾经依赖主题完全相等，容易出现“数据库中未找到原始邮件”。
- Streamlit 每次刷新会重跑页面，连接邮箱、查 API、查数据库时容易卡。
- 有些旧邮件没有新分类字段，导致模板路由仍走旧逻辑。

### 2.2 分类和模板命中还需要更严谨

- 不能因为附件叫 `Order Details.pdf` 就猜测是 Amazon。
- 必须先验证是 Amazon Purchase Issues，才走 Amazon 模板。
- 最终模板需要优先于普通 AI 生成和研发转发兜底。
- 错误模板必须能删除或停用，例如 Prime SELECT Firmware 这种错误步骤。

### 2.3 问题归并和研发汇报需要产品化

- 同类问题目前可以手动整理，但还需要更稳定的自动归并。
- 需要避免把“相似但不确定同根因”的问题强行合并。
- 报告必须包含具体用户邮箱，方便追溯和验证 AI 是否胡说。
- 研发报告要能区分：强证据、弱证据、相关线索、待确认线索。

### 2.4 控制台体验已经接近瓶颈

- Streamlit 适合当前试运行，但不适合长期正式客服工作台。
- 页面交互、筛选、详情、批量操作、异步任务、多人协作都会越来越吃力。
- 后端 FastAPI 可以保留，前端后续应迁移到 React + Vite + TypeScript。

## 3. 优先级计划

## P0：先保证售后邮件能被正确处理完

### P0-1. 稳定邮件草稿与原始邮件匹配

目标：控制台草稿区不再频繁出现“数据库中未找到原始邮件”。

需求：

- 读取草稿时保留 `In-Reply-To`、`References`。
- 优先用 `message_id` 匹配原始邮件。
- 主题匹配要忽略 `Re:`、`Fw:`、多余空格。
- 数据库匹配范围包括：
  - `drafted`
  - `forwarded_drafted`
  - `human_review`
  - `skipped`
  - `sent`
  - `failed_retry`
- 每类至少读取 500 条，避免旧草稿匹配不到。
- 匹配失败时显示可诊断信息：
  - 草稿主题
  - 草稿收件人
  - 是否有 `In-Reply-To`
  - 数据库候选数量

验收标准：

- 邮箱草稿页面能找到大多数原始邮件。
- 找不到时能说明原因，而不是只显示一句警告。

### P0-2. 修复旧邮件分类字段缺失问题

目标：旧邮件也能进入新的分类、问题归并和模板路由体系。

需求：

- 对历史 `drafted / human_review / skipped / failed_retry` 邮件补跑分类。
- 补齐字段：
  - `mail_category`
  - `issue_category`
  - `reply_template_category`
  - `classification_confidence`
  - `classification_reason`
  - `classification_evidence`
- 补跑时不覆盖人工已修正字段，除非明确选择强制重跑。
- 补跑结果要生成日志和统计。

验收标准：

- 旧邮件不再主要依赖 `ai_intent`。
- 模板路由优先使用新字段。

### P0-3. 最终回复模板优先机制

目标：研发给出方案后，后续同类邮件直接用 issue 的最终模板。

需求：

- 新邮件分类后，用 `product_model + issue_category / issue_signature` 查询 issue queue。
- 如果找到 `final_reply_template`：
  - 直接生成草稿
  - 标记为使用最终模板
  - 自动关联到对应 issue
  - 不再触发研发转发兜底
- 例如：
  - `GS1000 + gs1000_balance_output_issue`
  - 使用 `Issue #1` 的 Google Drive 固件模板
  - 不再一封封转发研发

验收标准：

- 同类问题再次出现时，草稿内容来自最终模板。
- issue 的用户数和邮件数继续增长。

### P0-4. 高风险模板和错误知识治理

目标：避免 AI 用错误步骤误导用户。

需求：

- 回复模板支持停用、删除、备注原因。
- 被删除模板不能再被 AI 使用。
- 高风险模板需要人工确认：
  - 固件升级
  - 恢复出厂
  - 按键组合
  - 拆机维修
  - 付费维修
- 记录模板使用日志：
  - 邮件 ID
  - 模板 ID
  - 模板分类
  - 命中原因

验收标准：

- Prime SELECT Firmware 这类错误模板不会再次出现。
- 能追溯每封草稿用了哪个模板。

## P1：把同类问题稳定归并，并能汇报研发

### P1-1. Issue Queue 归并规则升级

目标：把多封用户反馈归为同一个产品问题。

需求：

- 归并键优先级：
  1. 人工确认的 issue_signature
  2. `product_model + issue_category`
  3. AI 语义相似度
  4. 关键词兜底
- 归并时区分证据强度：
  - 强证据：同产品、同现象、同版本、多个用户
  - 中证据：同产品、相似现象，但版本未知
  - 弱证据：同产品、同大类，但表现不同
- 不确定时进入“待归并复核”，不能自动合并。

验收标准：

- `GS1000 - balance output issue after firmware update` 和 `GS1000-balance output issue` 能合并。
- `GE150 Pro Li 音频失真` 和 `GE150 Pro firmware/NAM 导入失败` 不被粗暴合并，只作为相关线索。

### P1-2. 研发报告生成

目标：客服能直接导出给研发的结构化报告。

报告必须包含：

- issue ID
- 产品型号
- 问题标题
- 问题分类
- 优先级
- 状态
- 影响用户数
- 关联邮件数
- 用户邮箱列表
- 邮件主题
- 时间
- 关键证据片段
- 附件信息
- AI 分类结果
- 证据强度
- 客服备注
- 需要研发确认的问题

报告格式：

- HTML
- Excel
- 用户邮箱 TXT

验收标准：

- 报告可以直接发给研发。
- 每条证据都能追溯到具体用户邮箱和邮件 ID。
- HTML 必须保存为真正 UTF-8，浏览器打开中文不乱码。

### P1-3. Issue 状态流转

目标：让客服知道每个问题当前卡在哪。

状态建议：

| 状态 | 说明 |
|---|---|
| 新发现 | 自动检测到，未确认 |
| 已确认 | 客服确认是有效问题 |
| 待复核 | 需要人工确认是否同类 |
| 已转研发 | 已发给研发 |
| 研发处理中 | 研发正在排查 |
| 已有方案 | 研发给出方案 |
| 批量回复草稿 | 已生成统一回复草稿 |
| 已批量回复 | 已回复关联用户 |
| 已关闭 | 问题闭环 |

验收标准：

- 控制台不显示内部英文状态码。
- 数据库仍保留稳定英文值，界面显示中文。

### P1-4. 研发方案回填

目标：研发给方案后，客服能把方案变成可复用模板。

需求：

- issue 详情页支持填写：
  - 解决方案摘要
  - 最终回复模板
  - 附件/链接
  - 适用产品
  - 适用固件版本
  - 注意事项
- 保存后：
  - issue 状态可改为“已有方案”
  - 后续同类邮件优先命中最终模板

验收标准：

- GS1000 balance output issue 已按此流程工作。

## P2：提升收信实时性和系统稳定性

### P2-1. IMAP IDLE + 轮询兜底

目标：邮件一进来尽快进入系统。

方案：

- 保留当前 60 秒 IMAP 轮询。
- 增加 IMAP IDLE 监听。
- IDLE 收到新邮件事件后立即触发一次处理。
- IDLE 断线后自动重连。
- 重连失败时继续使用 60 秒轮询兜底。

验收标准：

- 新邮件通常数秒内进入处理。
- 断线不漏邮件。
- 控制台显示：
  - 实时监听中
  - 轮询兜底间隔
  - 上次触发时间

### P2-2. Webhook 收信预研

目标：评估 Mailgun / SendCloud inbound webhook 是否作为长期收信方案。

注意：

- 不要直接改正式 `support@mooeraudio.com` 的 MX。
- 先使用子域名或测试地址：
  - `support-test@inbound.mooeraudio.com`
  - 或邮箱自动转发副本到 inbound 地址
- Webhook 入口必须校验签名。
- 初期只生成草稿，不自动发送。

验收标准：

- 测试邮箱邮件能通过 webhook 进入 FastAPI。
- webhook 邮件能复用现有分类、issue queue、模板逻辑。

### P2-3. 控制台性能优化

目标：减少 Streamlit 卡顿。

需求：

- 邮箱草稿数量缓存 30-60 秒。
- API 健康检查不要每次页面 rerun 都阻塞。
- issue 关联邮件只在选中 issue 后加载。
- 邮箱草稿读取提供手动刷新按钮。
- 数据库开启 WAL 和合理 busy timeout。

验收标准：

- 控制台切换页面明显更快。
- 后台邮件处理时，控制台不长时间卡死。

## P3：正式控制台迁移

### P3-1. 前端框架

建议方案：

- React
- Vite
- TypeScript
- TanStack Query
- TanStack Table
- shadcn/ui
- Tailwind CSS

原因：

- 后端已经是 FastAPI，不需要 Next.js 的服务端渲染。
- 控制台需要复杂表格、筛选、批量操作、局部刷新。
- React 生态更适合后台系统和长期维护。

### P3-2. 迁移顺序

第一批迁移：

1. 问题队列
2. 问题详情
3. 研发报告导出
4. 邮件草稿审核

第二批迁移：

1. 回复模板管理
2. 用户邮箱追溯
3. 知识库
4. 系统日志

Streamlit 保留为备用管理后台，直到正式控制台稳定。

## 5. 数据和安全要求

### 5.1 用户邮箱可追溯

所有报告必须包含具体用户邮箱，方便客服核查来源。

### 5.2 AI 不得无依据猜测

例如：

- `Order Details.pdf` 不能直接推断为 Amazon。
- 必须有明确 Amazon 字样、Amazon 订单号，或附件/正文中能验证 Amazon 来源，才走 Amazon Purchase Issues。

### 5.3 编码要求

所有 HTML 报告必须：

- 文件实际保存为 UTF-8。
- HTML 头部包含 `<meta charset="utf-8">`。
- 生成后验证中文没有变成 `?`。
- 避免 PowerShell 管道直接写中文报告。

### 5.4 密钥和配置

- 邮箱密码只放 `.env`。
- `config.yml` 不保存明文密码。
- `.env` 不提交。

## 7. MVP 验收标准

MVP 完成时，客服应能完成以下闭环：

1. 新邮件进入系统。
2. AI 给出分类和草稿。
3. 高风险邮件不自动跳过。
4. 同类问题进入 issue queue。
5. 客服能看到用户邮箱和证据。
6. 客服能导出研发报告。
7. 研发方案能写回 issue。
8. 后续同类问题自动使用最终模板。
9. 控制台能追溯每封邮件的分类、模板和处理状态。
