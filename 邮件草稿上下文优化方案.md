# 邮件草稿上下文优化方案

## 1. 背景

当前邮件草稿模块主要展示“当前这封邮件”的正文和 AI 生成的回复草稿。

这个模式可以处理单封邮件，但不适合长期作为团队客服系统使用。实际售后沟通往往是多轮往来，用户可能只回复一句：

```text
Any update?
```

如果系统只读取最近一封邮件，客服无法判断用户之前反馈了什么问题，也无法确认 AI 回复是否合理，只能回到邮箱里翻历史邮件。

这会带来几个问题：

- 客服仍然依赖原始邮箱，系统无法成为主工作台。
- AI 可能只基于当前邮件生成回复，缺少上下文。
- 团队成员接手时不知道前序沟通。
- Issue Queue、最终模板、用户历史无法形成完整闭环。

因此，邮件草稿模块需要升级为“会话上下文视图”。

## 2. 目标

客服打开一封草稿时，系统应展示完整上下文，而不是只展示最近一封邮件。

目标包括：

- 看到当前用户邮箱。
- 看到同一邮件线程的历史往来。
- 看到该用户过去相关邮件。
- 看到 AI 本次草稿。
- 看到之前系统或人工回复记录。
- 看到附件、订单、序列号等关键信息。
- 看到已关联的 Issue。
- 看到命中的模板或最终回复模板。
- 看到 AI 为什么这样回复。

最终效果是客服可以在控制台内完成判断、编辑和发送，不再频繁切回邮箱查来龙去脉。

## 3. 核心设计

### 3.1 从“单封邮件”升级为“会话线程”

系统需要为每封邮件建立线程关系。

优先匹配规则：

1. `Message-ID`
2. `In-Reply-To`
3. `References`
4. 规范化主题
5. 用户邮箱
6. 时间范围

其中规范化主题需要去掉常见前缀：

```text
Re:
RE:
Fw:
Fwd:
```

例如：

```text
Re: GE300Lite constant freeze after 5.02 upgrade
```

应归一为：

```text
GE300Lite constant freeze after 5.02 upgrade
```

### 3.2 草稿详情展示历史时间线

草稿详情页应增加“历史邮件时间线”。

示例：

```text
用户：ostitu2@gmail.com
当前问题：GE300 Lite - freeze after V5 / 5.02 firmware upgrade
关联 Issue：#55

时间线：
1. 2026-02-26 用户反馈 GE300 Lite 升级 V5 后 freeze
2. 2026-02-27 客服回复要求补充固件版本和视频
3. 2026-03-01 用户补充 V5.02 后仍然卡死
4. 2026-06-05 用户追问是否有新方案
5. 2026-06-10 系统生成最终模板草稿
```

每条时间线至少包含：

- 邮件 ID
- 发件人邮箱
- 收件时间
- 主题
- 邮件方向：用户来信 / 客服回复 / 系统草稿
- 邮件状态
- 正文片段
- 附件信息
- AI 分类结果
- 关联 Issue
- 使用模板

### 3.3 增加客户历史视图

除了同一主题线程，还需要按用户邮箱查看历史。

例如：

```text
客户历史：
- 2026-02-26 GE300 Lite V5 freeze
- 2026-03-01 跟进确认
- 2026-06-05 再次询问 any news?
```

这样可以解决用户换主题、转发、开启新邮件继续追问的问题。

### 3.4 增加 AI 上下文摘要

历史邮件可能很多，客服不应该每次都读完整线程。

系统应生成简短摘要：

```text
上下文摘要：
该用户自 2 月起反馈 GE300 Lite 升级 V5 后频繁 freeze。
此前客服已要求补充固件版本，并将问题归入 Issue #55。
当前邮件是用户询问是否有更新。
```

摘要必须带证据来源：

```text
证据：
- 2026-02-26 用户提到 "screen froze after the V5 update"
- 2026-06-05 用户追问 "Any news?"
```

这样客服可以快速判断 AI 是否在胡说八道。

### 3.5 AI 回复生成接入线程上下文

AI 生成草稿时，不应只读取当前邮件。

应输入：

- 当前邮件正文
- 同主题历史邮件
- 用户历史摘要
- 已关联 Issue
- 已确认最终回复模板
- 之前是否已经回复过类似内容
- 当前邮件是否只是追问、补充证据或确认信息

例如用户只发：

```text
Any update?
```

AI 应能根据线程上下文判断这是在追问：

```text
GE300 Lite / GE300 系列 V5 或 5.02 升级后 freeze
```

并优先使用 Issue #55 的最终回复模板。

## 4. 数据结构建议

### 4.1 emails 表新增字段

建议增加：

```text
message_id
in_reply_to
references
normalized_subject
sender_email
thread_key
direction
reply_to_email_id
```

字段说明：

| 字段 | 说明 |
|---|---|
| `message_id` | 邮件原始 Message-ID |
| `in_reply_to` | 当前邮件回复的是哪封邮件 |
| `references` | 邮件客户端提供的历史引用链 |
| `normalized_subject` | 去掉 Re/Fw 后的主题 |
| `sender_email` | 提取后的纯邮箱地址 |
| `thread_key` | 系统生成的线程标识 |
| `direction` | `inbound` / `outbound` / `draft` |
| `reply_to_email_id` | 回复对应的内部邮件 ID |

### 4.2 conversation_threads 表

建议新增线程表：

```text
conversation_threads
```

字段建议：

| 字段 | 说明 |
|---|---|
| `id` | 线程 ID |
| `thread_key` | 线程唯一键 |
| `customer_email` | 用户邮箱 |
| `normalized_subject` | 规范化主题 |
| `first_email_at` | 首封邮件时间 |
| `last_email_at` | 最新邮件时间 |
| `email_count` | 邮件数量 |
| `linked_issue_id` | 关联 Issue |
| `summary` | AI 上下文摘要 |
| `summary_updated_at` | 摘要更新时间 |
| `status` | open / pending / closed |

### 4.3 customer_profiles 表增强

现有客户表可以继续扩展：

```text
customer_profiles
```

建议字段：

| 字段 | 说明 |
|---|---|
| `email` | 用户邮箱 |
| `name` | 用户名 |
| `first_seen_at` | 首次来信时间 |
| `last_seen_at` | 最近来信时间 |
| `issue_count` | 涉及问题数量 |
| `latest_issue_id` | 最近关联 Issue |
| `tags` | 标签 |
| `notes` | 人工备注 |

## 5. 接口建议

### 5.1 获取邮件线程

```http
GET /api/v1/emails/{email_id}/thread
```

返回内容：

```json
{
  "email_id": "123",
  "thread_key": "ostitu2@gmail.com::ge300lite-constant-freeze-after-5-02-upgrade",
  "customer_email": "ostitu2@gmail.com",
  "linked_issue": {
    "id": 55,
    "title": "GE300 Lite - freeze after V5 / 5.02 firmware upgrade"
  },
  "summary": "...",
  "items": []
}
```

### 5.2 获取客户历史

```http
GET /api/v1/customers/{email}/history
```

返回该用户所有历史邮件、关联 Issue 和处理状态。

### 5.3 刷新线程摘要

```http
POST /api/v1/emails/{email_id}/thread/summary
```

用于手动重新生成 AI 上下文摘要。

## 6. 控制台展示方案

### 6.1 Streamlit 过渡版本

先不做复杂前端，当前 Streamlit 可以先加几个折叠区：

```text
当前邮件
AI 回复草稿
上下文摘要
历史邮件时间线
客户历史
关联 Issue / 模板命中原因
```

建议默认展开：

- 当前邮件
- AI 回复草稿
- 上下文摘要

默认折叠：

- 历史邮件时间线
- 客户历史
- 附件详情
- AI 判断证据

### 6.2 正式控制台版本

未来 React 控制台建议采用三栏布局：

```text
左侧：会话时间线
中间：当前邮件 + 回复编辑器
右侧：用户档案 / Issue / 模板 / 证据
```

这样更接近正式客服工作台。

## 7. 开发顺序

### 阶段 1：最小可用版本

目标：客服不用回邮箱查大部分上下文。

开发内容：

1. 提取并保存 `sender_email`。
2. 生成 `normalized_subject`。
3. 用 `sender_email + normalized_subject` 查询最近 20 封相关邮件。
4. 草稿详情页显示历史邮件时间线。

验收标准：

- 打开草稿时能看到同用户、同主题的历史邮件。
- 用户只回复一句追问时，客服能在系统里看到之前的问题。

### 阶段 2：标准线程匹配

目标：提高线程识别准确度。

开发内容：

1. 保存 `message_id`。
2. 保存 `in_reply_to`。
3. 保存 `references`。
4. 建立 `thread_key`。
5. 新增 `/api/v1/emails/{email_id}/thread` 接口。

验收标准：

- 同一邮件线程能稳定串起来。
- `Re:` / `Fw:` / 转发后的主题也能尽量归并。

### 阶段 3：客户历史视图

目标：支持团队接手和跨主题追踪。

开发内容：

1. 按邮箱查询客户历史。
2. 展示历史 Issue。
3. 展示历史模板使用记录。
4. 支持人工备注和客户标签。

验收标准：

- 客服能看到某用户过去所有问题。
- 不同客服接手时不用重新翻邮箱。

### 阶段 4：AI 上下文摘要

目标：减少客服阅读历史邮件的时间。

开发内容：

1. 基于线程邮件生成摘要。
2. 摘要附带证据片段。
3. 摘要缓存到数据库。
4. 支持手动刷新摘要。

验收标准：

- 摘要能说明用户问题来龙去脉。
- 摘要包含可追溯证据。
- 客服能判断 AI 是否乱总结。

### 阶段 5：AI 回复生成接入上下文

目标：让 AI 草稿真正理解多轮对话。

开发内容：

1. 回复生成 prompt 加入线程摘要。
2. 加入关联 Issue。
3. 加入最终模板匹配结果。
4. 加入历史已回复内容，避免重复问同样问题。

验收标准：

- 用户只回复 “Any update?” 时，AI 能识别是在追问哪个问题。
- 已有最终模板的问题优先使用最终模板。
- AI 不重复索要用户已经提供过的信息。

## 8. 风险和注意事项

### 8.1 不能只靠主题匹配

同一个主题可能被用户转发给不同人，或者不同用户使用相似主题。

所以主题匹配必须结合：

- 用户邮箱
- 时间范围
- Message-ID 链
- 产品型号
- Issue 指纹

### 8.2 AI 摘要必须可追溯

摘要不能只给结论，必须保留证据片段和来源邮件 ID。

否则客服无法判断 AI 是否胡说八道。

### 8.3 历史邮件不要一次性全部塞给 AI

长线程应先压缩成结构化摘要，再传给 AI。

否则成本高、速度慢，也容易让模型混淆。

### 8.4 回复前仍需人工确认

在系统稳定前，即使 AI 使用了上下文，也应先生成草稿，不直接自动发送。

## 9. 推荐下一步

当前最推荐先做阶段 1：

```text
sender_email + normalized_subject + 最近 20 封历史邮件时间线
```

原因：

- 改动小。
- 不需要马上重构数据库。
- 能立刻解决客服需要回邮箱查历史的问题。
- 后续可以自然升级到 Message-ID 线程匹配和 AI 摘要。

这个阶段完成后，邮件草稿模块就会从“AI 草稿列表”开始变成真正的“客服会话工作台”。
